# 설계 원칙 (njz 워크북)

이 저장소는 정적 단일 페이지 워크북 앱이다. `content.js`(데이터: `COURSE.weeks`)와
`app.js`(엔진/렌더)가 분리돼 있고, 주차는 `weeks` 배열에 객체를 추가/수정해 확장한다.
`njz-project`가 **라이브 배포본**, `njz-workbook/app/`이 **소스 미러**다.

아래 원칙은 모든 신규/수정 작업에 적용한다.

## 1. 선택지 우선, 자유서술 최소화 (가장 중요)
참가자들은 **퇴근 후 저에너지 상태**로 온다. 타이핑은 부담이다.
- 선택지로 환원 가능한 질문은 **`choices`(또는 `cardFilter`)** 로 만든다. 타이핑 0을 지향.
- 자유서술(`text`)은 **본질이 글쓰기인 것만** 남긴다 — 확언(`manifest`), 편지, 인생 한 줄(`journey`),
  꼭 직접 써야 의미 있는 성찰 1~2개.
- **한 주차의 필수 `text`는 2개 이하.** 나머지 `text`는 반드시 `optional: true`.
- 강제선택(둘 중 하나로 압박해 통찰을 끌어내는 문항)은 `multi: false`.

## 2. 화면엔 일상어, 구성개념은 프롬프트에만
- 사용자에게 보이는 문구는 쉬운 일상어. 심리학 용어·이론(SDT, 애착, 강점도메인 등)은
  화면에 노출하지 않는다.
- 구성개념은 **`promptForge`의 `system`/`collect` 라벨**에만 담아 AI에게 전달한다.

## 3. 엔진 재사용·최소 수정
- 가능하면 기존 컴포넌트(`text` `choices` `cardFilter` `sliders` `assess` `mandala`
  `promptForge` `reframe` `manifest` `journey` `digest` `progress` 등)를 재사용한다.
- 새 컴포넌트는 꼭 필요할 때만 추가하고, 저장 형태를 기존과 호환되게 한다
  (예: `assess`는 `sliders`와 같은 `{라벨: 0~10}` 맵을 저장해 `radarInner`를 공유).

## 4. 주차 구조
- 주차는 `weeks` **배열 순서로 렌더**되고 `id`로 네비게이션된다. `id`/step id는
  표시 번호(`no`/`badge`)와 **무관한 저장키**다.
- 주차를 삽입할 때는 **새 `id`로 추가하고 기존 주차의 `no`/`badge`만 갱신**한다.
  기존 step id는 절대 바꾸지 않는다(저장 데이터·교차참조 보존).

## 5. 배포 규칙
- 화면에 영향 주는 변경 시 `index.html`의 캐시버스터 `?v=N`을 올린다.
- `njz-project`(라이브)에 먼저 반영하고 `njz-workbook/app/`에 동일 파일을 동기화한다.
- 매 변경은 양 레포에 커밋·푸시하고 드래프트 PR을 연다.
